| Property | Value |

## Spacing Scale

This phase has no traditional page/card spacing surface (it is gesture + text-tree rendering, not layout). The one spacing-relevant contract is **fold indentation**:

| Token | Value | Usage |
|-------|-------|-------|
| fold-indent | 16px (1 tab-stop at the field-tree's monospace advance width) | Each rank-1 reveal indents one level under its parent row (`object_exploration.md` §5, §13: "the inline rank-1 fold indents under the parent with `--silver-900` guide lines") |
| row-gap | 4px | Vertical gap between sibling field-tree rows (matches existing `field_tree.md` §10 line-height 1.5 at 13px) |

Exceptions: none. No new spacing scale is introduced; indentation rides the existing monospace grid (`field_tree.md` §4: "nesting by tab depth").

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Field-tree print (key/value) | 13px | 400 (regular; no bold weight exists in this surface) | 1.5 |
| Type slot (`: int`, `: WebDriver`, `: {FunctionOutputType}`) | 13px (same as field-tree; no size step) | 400, rendered in `--text-dim` outline | 1.5 |
| `{ref}` token | 13px (inline with field-tree) | 400, `--silver-700` underline | 1.5 |
| Read-only / 🔒 label | 13px | 400, `--text-lock` (`--silver-900` outline) | 1.5 |

Source: `field_tree.md` §10 + `theme.md` §5 ("Monospace is mandatory... 13px, line-height 1.5"). **Only one font size (13px) and one weight (400) exist in this surface** — this is intentional, not an omission: the black-core+silver-outline system encodes hierarchy through outline intensity (`--silver-100..900`) and indentation, never through size/weight steps. Do not introduce a second size or a bold weight for this phase's new gesture affordances (hover preview, fold guide, brace states) — reuse the existing outline-intensity vocabulary.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--black` `#000000` | Every panel/node fill, the editor canvas, the void — see `theme.md` §2.1 ("There is one fill, and it is black") |
| Secondary (30%) | `--silver-700` / `--silver-800` outline tiers | Resting borders, hairline dividers, fold guide lines, soft-link strokes |
| Accent (10%) | `--silver-100` / `--silver-300` outline tiers | RESERVED FOR: hover-brighten on a token's outline (the single most-used affordance, `theme.md` §4); active/focus 2px outline on an open edit field; the specular caret in an edit textarea. NOT a fill color — silver never fills, only outlines. |
| Destructive | `--accent-error` `#b25b5b` (already in `styles.css`) | Reserved for: a broken `{ref}` underline, and (NEW in this phase) the double-right-click delete confirmation flash on the target token before removal — see Copywriting below |

**No new colors are introduced by this phase.** The three brace render states (braced-hidden / revealed-internal / resolved-external, D-04) are **not** color-coded — they are distinguished by braces-present-or-absent + indentation + link-style (dashed-never, solid-only), per `object_exploration.md` §5.1 and the CLAUDE.md forbidden-concepts list ("Stray dotted UI lines... removed"). The resolved-external state's solid link reuses Phase 6's `--accent-arrow` 2D↔3D substrate ONLY when the resolved target is a 3D-backed node; a resolved-external link between two 2D-only nodes is a plain `--silver-300` solid line (no arrow color) — this is a genuinely unspecified seam, flagged as **Assumption A1** below.

Accent reserved for: hover-outline-brighten, active-edit-outline, edit-caret. Never a badge, never a fill, never a status color (per `theme.md` §2.4: "Provenance, rank, status, and type are position, outline intensity, glyph, and tooltip text — never a colour badge or fill").

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | None — this phase has no buttons/CTAs; it is gesture-driven (hover/click/drag/right-click). The closest analogue is the **double-right-click delete** action, which has no confirmation dialog (per the seven-gesture model, deletion is immediate — see Destructive below) |
| Empty state heading | Not applicable as a heading; an unbound `{}` (singular empty braces) IS the empty state — "a spaces-bearing key beside a singular empty `{}`" (`field_tree.md` §3, the `{}` on-ramp for a new unbound ref slot). No prose empty-state copy exists in this surface; the empty state is structural (an empty brace token), consistent with the no-chrome contract. |
| Empty state body | N/A — see above. A bound-but-unresolved function output renders blank (e.g. `dom {}` in the DuckDuckGo walkthrough) rather than placeholder prose — this is intentional per `object_exploration.md` §5.1 ("outputs blank until bound or inverse-referenced"). Do not add hint text inside the braces. |
| Error state | A broken `{var}` reference renders with an `--accent-error` underline in place of the normal `--silver-700` underline (`field_tree.md` §10); no error toast/banner — the error IS the visual state of the token itself. A failed gesture (e.g. a rejected drag-wire onto a type-incompatible target) surfaces via the gateway's existing `--accent-error` border-flash on the offending control (`gesture_gateway.md` §10) — no copy string is shown beyond the existing tooltip mechanism. |
| Destructive confirmation | Double-right-click delete: **no modal confirmation** — the double-click-vs-single-click distinction (delete vs fold) IS the confirmation gesture per the §18.32 non-collision design intent. On commit, the deleted token's row briefly flashes its outline to `--accent-error` for ~120ms (matching the existing `--accent-error` error-envelope treatment) before being removed from the DOM — a "this just happened" acknowledgment, not a "are you sure" gate. This is **Assumption A2**: the source docs (`object_exploration.md` §4, `gesture_gateway.md` §5.6) specify the gesture and its REST/mirror wiring but not an explicit visual delete-acknowledgment; the 120ms `--accent-error` flash is proposed here as the minimal non-blocking signal consistent with theme.md's error-outline vocabulary, and does not change the no-modal contract. |

No prose strings are net-new to this phase beyond the assumptions flagged above — the design is deliberately copy-free (structural/gestural, not textual) per the black-slate no-chrome contract.

## Interaction Contract — The Seven-Gesture Model (the core of this phase)

Source: `object_exploration.md` §4 + §18.32 non-collision rules; `07-CONTEXT.md` Locked-by-design. This table is the load-bearing artifact of this UI-SPEC — planner/executor must implement exactly these seven gestures with exactly this disambiguation.

| # | Gesture | Target | Effect | Disambiguation (§18.32) |
|---|---------|--------|--------|--------------------------|
| 1 | **Hover** | a typed field / `{token}` (within its mini-card box bounds) | Transient next-rank preview; un-hover collapses unless committed by right-click | Distinct from right-click-commit by persistence only — hover is ephemeral, right-click is sticky |
| 2 | **Single left-click** | a token (key / type / value) | Edit as a blended, borderless, cursor-blinking field — no per-token chrome | Targets the token's *interior* (the editable text region); right-click on the same token targets fold, not edit — same button-area, different button |
| 3 | **Right-click** | a token (key/type/value, base or reference) | Toggle expand/collapse over rank-1 links, inline in panel form only; preserves per-subtree fold state | A right-click on a CHILD token folds that child; a right-click on the BASE/self node (`self:WebBrowser`) collapses the whole panel — same button, target disambiguates |
| 4 | **Right-click** | the base/self node | Collapse entire panel to the singular self node; re-expand restores prior fold state | See #3 |
| 5 | **Double left-click** | panel body / base node (panel form) OR a graph node (graph form) | panel→graph / graph-node→containment-panel; symmetric in both directions | Targets the panel body / base region OUTSIDE any editable token — never races a single-left edit |
| 6 | **Left-click-drag** | from one node onto another (graph form only) | Wires a link; target inherits source's I/O types + object model; duplicate-instantiates source as rank-1 component | A drag (press→move→release between nodes) is distinct from a click (press→release, no move) — drag wires, click edits |
| 7 | **Double-right-click** | a token reference/instance in panel or graph form | Deletes that reference/instance | Distinct from single-right-click (rank-1 fold) by click-count, same button |

**Read-only python-native nodes (🔒):** single-left is a no-op highlight (edit refused); hover/right-click/double-left ALL still work — exploration is always permitted on read-only nodes, only editing is blocked.

**Implementation note for executor:** gestures #2/#3/#7 all target the same token DOM element via different mouse buttons/click-counts — the existing `magic_markdown_gestures.mjs` dispatch table (currently has `TOGGLE_FOLD` and `COLLAPSE_TO_NODE`) needs a double-right-click delete action and a drag-state machine added; do not fork a separate token-rendering path per gesture.

## Type-Slot Row Rendering Contract

Source: `object_exploration.md` §2/§5; `field_tree.md` §4.

**Raw object inspection panels** (e.g. hovering `driver: WebDriver`) — full typed form:
```
key : Type = value
```
- `key` — `--text-primary` outline (black glyph, `--silver-300` outline)
- `:` separator — `--text-dim` outline
- `Type` — `--text-dim` outline (the type slot recedes behind key/value, no colour)
- `=` separator — `--text-dim` outline
- `value` — `--text-primary` outline, OR a `{ref}` token with `--silver-700` faint underline

**User compute nodes** (e.g. `DuckDuckGo` / `{scanner}`) — rank-1 minimalism, type-stripped:
```
key
   {ref}
```
No `: Type` slot is rendered at all — purely structural over tab+newline (`object_exploration.md` §5.1 N.4/N.5). This is a hard distinction, not a styling toggle: raw object panels declare `key:Type=value`; user compute panels never show a colon-type pair anywhere in their tree, even when right-click-revealed.

**Indentation:** each rank-1 reveal indents its children one tab-stop (16px, per Spacing above) under the parent row, with `--silver-900` fold guide lines (felt, not seen — per `theme.md` §3 dividers).

## Three Brace Render States (D-04 — pulled into this phase)

Source: `object_exploration.md` §5.1 (final paragraphs) + §O.1a; `07-CONTEXT.md` D-04. One invariant graph link; three RENDERING states over it.

| State | Visual | Trigger | Panel↔Graph parity |
|-------|--------|---------|---------------------|
| **braced-hidden** | `{ref}` renders with braces, `--silver-700` faint underline, no children shown | Default state for any unrevealed reference; in graph form, hover previews (no commit) | Graph shows nothing for this ref until revealed in either form |
| **revealed-internal** | Braces drop; rank-1 fields unfold inline, indented under the parent (per Type-Slot contract above) | Right-click in panel form; click-to-instantiate in graph form | Revealing in panel reveals in graph (and vice versa) — node-count parity is mandatory, not advisory |
| **resolved-external** | `{ref}` resolves to a SOLID link (line, no dash, no arrowhead) to an already-visible node elsewhere in the same graph | Automatic, the moment the referenced node becomes visible by any other path (e.g. another panel revealed it first) | The link itself is the parity signal — both forms draw the same solid line |

**Hard constraints (do not violate):**
- NO dashed/dotted lines anywhere (CLAUDE.md Forbidden Concepts; `theme.md` §2.4: "links are undirected lines, no arrowheads — §O.16"). All link strokes are solid.
- The resolved-external solid link reuses the Phase 6 (REAL-04) 2D↔3D `--accent-arrow` yellow solid-line substrate ONLY for links that terminate at a 3D-backed node (`data-3d-node-id` present). A 2D-only resolved-external link (both ends are 2D panel/graph nodes) is `--silver-300` solid, 1–2px, no colour — this is **Assumption A1** (flagged above under Color): the source docs establish the resolved-external mechanic and the REAL-04 substrate's reuse for 3D-backed targets, but do not explicitly state the 2D-only case's stroke color; `--silver-300` (the existing "hard link" token per `link_layer.md`/`theme.md` §7) is the consistent default and does not require new tokens.
- Underlying graph link is identical across all three states — never duplicate the edge record per render state; this is a rendering toggle on one `ConceptEdge`, not three edge types.

## Fold-State Preservation Contract

Source: `object_exploration.md` §8; M.6.

- Fold state = the set of unfolded `node_path`s for a card: `ui.node_fold_state[card_id] = { expanded_paths: [...] }`.
- Collapsing a parent hides its subtree but RETAINS each descendant's fold flag.
- Re-expanding a parent restores descendants to exactly the fold levels they held before collapse.
- Right-clicking the base/self node collapses the whole panel while retaining the ENTIRE fold tree; re-expanding restores it intact.
- Anti-goal (§18.32): resetting folds to fully-collapsed on every toggle. The `panel-gesture-fold-roundtrip` env-scenario is the acceptance gate for this — verify against it, not a screenshot.

## Hover-Preview Contract

Source: `object_exploration.md` §4 (gesture 1), §5; `theme.md` §4.

- Hover on a typed field/`{token}` (within its mini-card box bounds) shows a TRANSIENT next-rank preview — the same rank-1 subtree that right-click would commit, but ephemeral.
- Visual signal for hover-ability: the token's silver outline brightens (`--silver-700` rest → `--silver-300` hover) — no fill tint, no colour change, per `theme.md` §4 state model.
- Un-hover (mouse leaves the token's bounds) collapses the preview UNLESS it was committed by a right-click in the same interaction — committed state persists independent of hover.
- This is the SAME mechanic in both panel form (inline preview) and graph form (`object_exploration.md` §5.1: "hover previews the rank-1 links and a click instantiates the rank-1 walk as visible nodes").

## Drag-Wire Affordance Contract

Source: `object_exploration.md` §4 (gesture 6); `07-CONTEXT.md` D-02 (port from `cp/edge_manager.js`); EXPLORE-03.

- Graph form only. A left-click-PRESS on a source node, followed by MOVE, followed by RELEASE on a target node, wires a link.
- During the drag: render a transient solid line (no dash) following the cursor from the source node to the current pointer position — this is the only "live" visual feedback during the gesture; no other chrome (no tooltip, no snap-zone highlight beyond the target node's own hover-brighten when the cursor is over it).
- On release over a valid target: the target node INHERITS the source's input-output types + object model (N.4) — this is a structural mutation (the target's rendered shape changes), not merely a new edge; the gateway fires `concept-edge-create`.
- On release over empty canvas (no target): the drag is discarded, no edge created, the transient line disappears with no error state (this is not an error — the user simply changed their mind).
- A drag (press→move→release between two nodes) must be distinguished from a click (press→release with no intervening move) — see Interaction Contract gesture 6 disambiguation. A move-threshold of a few pixels (implementation discretion — not design-locked) avoids accidental drag-classification of a slightly-jittery click.

| ID | Gap | Resolution proposed | Risk if wrong |

| A1 | Resolved-external link color when BOTH endpoints are 2D-only (no 3D backing) | `--silver-300` solid 1-2px (existing "hard link" token), no new color | Low — cosmetic only, easy to adjust post-hoc since no new token is introduced |
