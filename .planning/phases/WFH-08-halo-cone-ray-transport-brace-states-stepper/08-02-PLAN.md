<objective>
Realize HALO-03: the halo cone-ray transport (§O.18). Opening a halo TRANSPORTS retrieved 3D
nodes onto a SHARED CONE whose apex is the 2D query element's screen position; the backend's
normalized triple-product similarity sets each candidate's radial + along-ray distance (closer =
more similar); camera azimuth sets the angular placement; deleting a result transports the
next-most-similar in.

The hard math already exists server-side — `backend/api/routes.py::get_apparitions` (the
`transport=True` branch) computes `transport.{similarity, radial, along_ray}` from the real
triple-product (`pagerank · tfidf_cos · nomic_cos`). This plan's job is FRONTEND-ONLY wiring per
D-04: a NEW dedicated `fe/halo_cone.mjs` that consumes those backend values + the projector's
camera state, plus a new `projector.placeHaloCandidates` renderer. Per D10 the frontend NEVER
recomputes similarity.

Purpose: The first of the three Phase-8 requirements with 3D geometry — the cone is the surface
the brace states (08-01) and the stepper (08-03) compose around.
Output: `fe/halo_cone.mjs` (pure geometry) + `placeHaloCandidates` (projector render) + the
Wave-0 unit scaffold + a stub-backed cone-transport e2e.
</objective>

<resolved_open_questions>
**Open-Q1 (the cone along-ray 3D basis vector) — RESOLVED by reading `docs/DOMAIN_MODEL.md` §O.18
(line 1893) + `docs/frontend/halo.md` §3.2 (line 25) directly. The contract is:**

- **Apex** = the 2D query element's screen position (the focal stuck-node primitive, or one of its
  singular-field/graph primitives), read live each frame from the focal's bounding rect — exactly
  the `{cx, cy}` focal centre `haloLayout(focal, …)` already takes.
- **Per-candidate ray** = the ray FROM the apex TOWARD `screen-projection(candidate.world_pos)` —
  i.e. project the candidate's EXISTING 3D node world position (`projector.nodeWorldPosition(id)` →
  `projector.project(x,y,z)` → screen `{x,y}`) to screen, and draw the ray apex→that-screen-point.
  DOMAIN_MODEL §O.18 line 1893: *"for each projector-neighbour c_i: ray = focal_centre →
  screen-projection(c_i.world_pos); intersect ray with cone — collapsed-singular phantom placed at
  intersection."* The ray direction is NOT camera-forward and NOT scene-origin — it is the
  apex→projected-node-screen-direction (Assumption A1 in 08-RESEARCH is thereby RESOLVED to this
  third option).
- **Distance along that ray** = the backend's `transport.along_ray` (along the ray from apex) +
  `transport.radial` (the cone's perpendicular offset). Consume both VERBATIM — never recompute
  `(1-s)·R` from a raw score (D10).
- **Angular placement** = camera-azimuth-dependent: the candidate's projected screen direction
  already encodes camera view (orbiting the projector re-projects `c_i.world_pos`, rotating the
  ray), and the HSV phase locks to `azimuth()` per halo.md §3.3. Read `projector.azimuth()` for the
  HSV-phase lock; the angular ray direction comes for free from the live screen-projection.
- **For non-projector candidates** (a retrieval-similar node with NO 3D backing): fall back to the
  EXISTING 2D `haloLayout` polar placement (`angular = nomic-direction, radial = (1−norm_score)·
  r_extent`, §O.18 line 1895). The 2D path in `magic_markdown_halo.mjs` is NOT removed — the cone
  transport is ADDITIVE for candidates that DO have a backing 3D node.

This resolves 08-RESEARCH Open Question 2 / Assumption A1 and 08-PATTERNS Open Question 1.
