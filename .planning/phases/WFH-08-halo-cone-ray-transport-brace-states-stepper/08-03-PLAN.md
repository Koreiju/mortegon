<objective>
Realize STEP-01: the 2D per-sample stepper drives 3D focus ONE-WAY (§O.6/§O.7/§O.11). Advancing
`{chunk samples}` in 2D flies/highlights the corresponding chunk in 3D, while the 3D ALWAYS shows
the full per-sample distribution.

Two halves, both small:
1. **Backend (per D-03 — the ONE real gap):** today's signal-stream cursor carries
   `{card_id, total, signal_index, signal_id?, ...}` but NO caller ever populates `signal_id` with
   a real chunk concept_id (CONFIRMED in 08-PATTERNS by grepping every `set_signal_stream(` call
   site). Resolve `signal_id` server-side from the card's ordered sampled-chunk list, so the cursor
   always points at a real 3D chunk. This keeps data correlation server-side per D10 — the frontend
   never guesses which chunk maps to which index.
2. **Frontend:** a new `projector.flyToNode(nodeId)` + `highlightNode(nodeId)` (node-id-keyed
   siblings of the URL-keyed `frameCameraToRoot`), and a new `fe/stepper.mjs` driver that advances
   the cursor and focuses the 3D node. The 3D NEVER drives the 2D back.

This plan modifies `projector.mjs` (adding flyToNode/highlightNode), which 08-02 also touches
(placeHaloCandidates) — hence Wave 3, strictly after 08-02 (same-file dependency).

Purpose: The third Phase-8 requirement; the first time the 2D iteration affordance has a 3D
consequence.
Output: server-side chunk-id resolution; flyToNode/highlightNode; `fe/stepper.mjs`; pytest + unit
+ e2e + an extended env-scenario.
</objective>

<resolved_open_questions>
**Open-Q1 (chunk-id resolution: backend vs frontend) — RESOLVED to BACKEND-SIDE, populating the
EXISTING `signal_id` field.** The `signal_id` key ALREADY exists in the signal-stream entry shape
(`ui_state_service.py` set_signal_stream line 689/713, advance_signal preserves it via the
`entry = dict(...)` copy) but no caller ever sets it to a real chunk concept_id (CONFIRMED in
08-PATTERNS line 120 by grepping every `set_signal_stream(` call site in rollout_coordinator.py).
The fix: thread an `ordered` chunk-id list (the card's `sampled_chunks`, the same ordered list the
spine mirror already stores per §17.14) into set_signal_stream/advance_signal and resolve
`signal_id = ordered[signal_index]` server-side. Per D10 this keeps the index→chunk correlation in
the backend; the frontend reads `state.signal_stream[cardId].signal_id` and never indexes into a
client-side array (which could silently desync from `signal_index` ordering during live streaming —
08-RESEARCH Open Question 1's stated risk). Reuse the existing `signal_id` key — do NOT add a
parallel `chunk_id` field (one source of truth).
</resolved_open_questions>
