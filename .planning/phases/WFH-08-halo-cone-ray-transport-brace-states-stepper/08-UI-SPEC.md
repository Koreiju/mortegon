## Spacing Scale

This phase has no traditional page/card spacing surface — it is 3D-geometry placement (cone transport), gesture-driven render-state toggling (brace states), and one-way focus-drive (stepper), not 2D layout. The 8-point scale is **not applicable** to the cone (which is parametric polar/conic geometry, not a CSS grid) or to the stepper (which drives camera/highlight state, not DOM position). The one spacing-relevant carryover:

| Token | Value | Usage |
|-------|-------|-------|
| fold-indent | 16px (Phase 7, unchanged) | revealed-internal `{ref}` children inherit the EXISTING fold-indent contract — one tab-stop under the parent row, per `07-UI-SPEC.md` Spacing. HALO-04 does not introduce a second indentation rule; it WIRES the existing classification into the existing indent. |
| row-gap | 4px (Phase 7, unchanged) | vertical gap between sibling field-tree rows, including the newly-visible `▸`/`▾`-prefixed rows |

Cone geometry parameters (apex/radial/along-ray/angular) are NOT a spacing-scale concept — they are documented in the Interaction Contract below as geometry, not pixels-on-a-grid. Exceptions: none beyond the above carryover.

## Interaction Contract A — HALO-03 Cone-Ray Transport (§O.18 / `halo.md` §3.2)

Source: `halo.md` §3.2, `08-CONTEXT.md` Locked-by-design, `projector.md` §4/§5. This is the load-bearing geometry contract for this surface.

**Apex/anchor:** the cone's apex is the **2D query element's screen position** — the focal stuck-node primitive, or one of its singular-field/graph primitives (`halo.md` §1, §O.19). This is read live each frame from the focal's bounding rect (mirroring the existing `haloLayout(focal, …)` signature in `magic_markdown_halo.mjs`, which already takes a live `{cx, cy}` focal centre).

**Placement rule (frontend renders, never computes):**

| Axis | Rule | Source of the value |
|------|------|----------------------|
| Radial + along-ray distance | Set by the **normalized triple-product similarity** (`pagerank · tfidf_cos · nomic_cos`); MORE similar → NEARER the apex. Monotonic, not linear-required — the e2e gate asserts ordering, not a specific curve. | Backend retrieval index (`/apparitions?transport=1&ray_project=1`); the frontend reads `similarity` per candidate and applies the EXISTING `r = rBase + (1 - sim) * rExtent` formula already in `haloLayout` (no new math needed if the cone reuses the same monotonic mapping in 3D) |
| Angular placement | Set by **camera azimuth/view** — the projected line of the node's intersection along the shared cone's surface normal | The projector's existing `camera_azimuth_phase` (already driving HSV rotation, `projector.md` §5.3) — HALO-03 reads this same live value, does not introduce a second camera-angle source |
| Node identity at each cone position | Each transported node is a **collapsed singular node** (mode `collapsed`/`phantom` per `concept_view.md` §2), carrying its image billboard (via the existing `TextureCache`) OR its HSV colour from the 6-vector | `concept_view.md` modes table; `projector.md` §9 `TextureCache` — reused verbatim, no new node representation |

**Delete-and-replace:** deleting a halo result (the candidate occupying a cone position) transports the **next-most-similar** candidate from the retrieval queue into the vacated position, dynamically re-rendering (`halo.md` §3.2, §O.14). This is a re-rank, not a destructive action — see Copywriting above. No animation timing is design-locked; the planner/executor may choose an interruptible tween consistent with the projector's existing tween discipline (`liveness.md` §2, already governing `frameCameraToRoot`/chunk position tweens) — **this is Assumption A4** (flagged below): the source docs specify WHAT re-populates, not HOW fast/eased the transport animates.

**Stroke contract (hard constraint, no exceptions):** every cone ray / transport line is **SOLID** — no `stroke-dasharray` anywhere (CLAUDE.md Forbidden Concepts: "Stray dotted UI lines... removed"; `theme.md` §2.4: "links are undirected lines, no arrowheads"). This reuses the EXISTING halo ray rendering in `magic_markdown_halo.mjs` (`haloVDom`'s `<line>` elements already carry no `stroke-dasharray`) — HALO-03 does not need to add a no-dash rule, it needs to NOT regress the existing one when extending 2D ray placement into 3D cone placement.

**Reuse boundary (D-04):** the cone transport extends/wraps the EXISTING `magic_markdown_halo.mjs` (`haloLayout`/`haloVDom`) for the 2D ray-projection half and reads `projector.mjs`'s `project()` (`{x,y,inFront,ndcZ}`) + camera azimuth for the 3D half. A new dedicated module (e.g. `fe/halo_cone.mjs`) is the PREFERRED seam per D-04 — it should consume both existing modules' outputs rather than duplicating ray math or camera-state tracking.

## Interaction Contract B — HALO-04 Three `{ref}` Render States (§O.1a / D-02)

Source: `08-CONTEXT.md` D-02, Phase 7's `07-UI-SPEC.md` "Three Brace Render States" table (REUSED VERBATIM — this phase WIRES the classification into the renderers, it does not re-derive the states). `magic_markdown.mjs`'s `classifyBraceStates` (lines 259-281) ALREADY computes this correctly per-line; the gap closed by this phase is purely that `panelVDom`/`graphVDom` (in `magic_markdown.mjs`/`magic_markdown_panel.mjs`) currently IGNORE `line.braceState`.

| State | Visual (unchanged from Phase 7) | Wiring gap this phase closes |
|-------|--------|-------------------------------|
| **braced-hidden** | `▸` glyph + literal `{ref}` with braces retained, `--silver-700` faint underline, no children shown | `panelVDom` must render the `▸` glyph (currently rendered without state-aware glyph selection) |
| **revealed-internal** | `▾` glyph + braces drop + rank-1 fields unfold inline, indented one fold-indent (16px) under the parent | `panelVDom`/`graphVDom` must render the `▾` glyph AND the inline children keyed off `line.braceState === BRACE_REVEALED_INTERNAL`, not off a separate ad hoc glyph check |
| **resolved-external** | A SOLID cross-ref link (no dasharray, no arrowhead UNLESS 3D-backed) to the already-visible target node elsewhere in the same graph | `graphVDom` currently has **no drawing path at all** for this state (07-03 SUMMARY's self-flagged gap: "resolved-external solid link has no graphVDom drawing") — this phase adds that SVG/DOM line draw, reusing the `#link-layer` solid-line idiom from `magic_markdown_panel.mjs` (the SAME substrate Phase 6's REAL-04 2D↔3D arrow uses) |

**Glyph contract (NEW in this phase — the visual realization of the existing classification):**

| `line.braceState` value | Rendered glyph | Glyph color/weight |
|---|---|---|
| `BRACE_HIDDEN` | `▸` (closed triangle, pointing right) | `--silver-700` outline, same weight as surrounding field-tree text — NOT a separate icon font, a plain Unicode glyph inline in the monospace print |
| `BRACE_REVEALED_INTERNAL` | `▾` (open triangle, pointing down) | `--silver-300` outline (brighter — matches the "open" affordance state already used for hover-brighten elsewhere, `theme.md` §4) |
| `BRACE_RESOLVED_EXTERNAL` | No glyph on the referring line itself — instead, a solid line is drawn from this line's position to the already-visible target node | Link stroke color per the Color section above (`--accent-arrow` if 3D-backed target, `--silver-300` if 2D-only) |

**Node-count parity (mandatory invariant, asserted by e2e):** revealing a `{ref}` in panel form reveals the SAME rank-1 children in graph form, and vice versa — both forms read the SAME `line.braceState` classification over the SAME underlying `ConceptEdge`. The underlying graph link is identical across all three render states; this is a rendering toggle, never three edge types (`08-CONTEXT.md` D-02, Locked-by-design). The two-level ref-chain e2e (ROADMAP success criterion 2) is the acceptance gate for this parity — verify against it, not a screenshot.

**Hard constraints (do not violate, unchanged from Phase 7):**
- NO dashed/dotted lines anywhere — all link strokes solid.
- The resolved-external link reuses Phase 6's `#link-layer` / `--accent-arrow` 2D↔3D substrate ONLY for 3D-backed targets (Assumption A1, Phase 7, carried forward unchanged).
- Braces/glyphs are NOT color-coded by themselves beyond the outline-intensity shift (`--silver-700` resting → `--silver-300` open) — no fill, no badge.

## Interaction Contract C — STEP-01 2D→3D One-Way Stepper (§O.6/§O.7/§O.11)

Source: `08-CONTEXT.md` D-03, `projector.md` §1 ("Advancing the 2D per-sample stepper drives the 3D focus (one-way 2D→3D)"), `field_tree.md` §7 (the existing signal-stream gating this stepper reads).

**One-way contract (hard constraint):** the 2D `{chunk samples}` stepper DRIVES 3D focus; the 3D **NEVER** drives the 2D stepper back. This is asymmetric by design, not a missing feature — do not add a 3D-click-advances-2D-stepper gesture in this phase (out of scope; would violate the one-way invariant).

| Stepper action | 2D render (existing, unchanged) | 3D render (this phase wires) |
|---|---|---|
| Advance to sample N | `FieldTree` swaps the ONE visible print form in place (`field_tree.md` §7: "renders only the current signal element... and nothing else") — no "iteration 3 of 12" overlay | The projector flies/highlights the camera/visual focus to the corresponding 3D chunk node — a fly-to-tween (reusing the EXISTING `frameCameraToRoot`-style interruptible tween discipline) and/or an outline-brighten highlight (`--silver-300`, the same accent-reserved affordance used elsewhere, NOT a new color) |
| 3D state at all times | N/A (2D shows only the current sample) | The 3D **ALWAYS** shows the **FULL per-sample distribution** — every sample-node stays rendered and positioned; the stepper highlights ONE member, it never subsets/hides the others (`08-CONTEXT.md` D-03: "the stepper only MOVES FOCUS... it never subsets the 3D distribution") |

**Highlight visual (Assumption A5, flagged below):** the source docs establish that 3D "flies/highlights" the corresponding chunk but do not pixel-spec the highlight's exact visual treatment beyond "passive physical + HSV rotation" being the existing liveness signal for nodes without billboards (`projector.md` §10, carried from §P.4/CASC-02 — though CASC-02 itself is Phase 9 scope, the existing passive-rotation idiom is already live in the projector). Proposed treatment: an outline-brighten (matching the `--silver-300` accent-reserved hover affordance) applied to the focused node's existing HSV/image fill, NOT a new color or a pulsing animation — consistent with "accent is outline-only, never a new hue." This is non-blocking; the planner may also choose camera-fly-only (no outline) if that satisfies the e2e's "flies/highlights" assertion more simply.

**Backend plumbing (D-03):** frontend-first, reusing `signal_stream`/`signal_advance` (Phases 4/§R) and the projector's existing fly-to/highlight primitives. New backend signal/focus plumbing is in-budget ONLY where the existing surface does not already expose what the one-way drive needs — this is NOT license to move layout/retrieval computation into the frontend (D10 still holds).
