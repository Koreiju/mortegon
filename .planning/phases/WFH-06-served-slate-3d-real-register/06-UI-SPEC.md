# Phase 6 — UI Design Contract

> Visual and interaction contract for the served `fe/` projector porting the 3D Real
> register (force-directed UMAP-radial layout, per-URL placement, image billboards,
> solid 2D↔3D arrows). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Framing note (load-bearing):** this is a brownfield PORT, not new design. Every
> value below is already locked by `docs/frontend/projector.md`, `billboard.md`,
> `link_layer.md`, `theme.md`, the legacy `cp/force_layout.js` reference
> implementation, and `CLAUDE.md`'s Forbidden Concepts. This document is the
> CODIFICATION of those contracts into a form the planner/executor/checker can
> verify line-for-line — it does not introduce new visual decisions. There are
> no conventional forms/buttons/CTAs in this surface; "UI" means 3D geometry,
> billboard appearance, color mapping, link-arrow style, and camera behavior.

## Spacing Scale

Not applicable in the conventional 2D sense — there is no padding/margin/gap system inside the WebGL viewport. The "spacing" contracts for this phase are 3D-geometric, declared exactly (no CSS box model applies):

| Token | Value | Usage |
|-------|-------|-------|
| `NODE_RADIUS` | `0.9` (world units) | Collider radius per chunk node (text or image billboard) — shared between both per REAL-03/REAL-04 collider-sharing requirement (`force_layout.js` line 160; `theme.md`/`projector.md` §5.4) |
| `COLLIDER_SAFETY` | `1.4` | Safety multiplier on minimum separation (`force_layout.js` line 161; `projector.md` §5: "`COLLIDER_SAFETY ≥ 2.0`" — **note the legacy code ships `1.4`; see Assumption A1 below**) |
| `MIN_SEPARATION` | `2 × NODE_RADIUS × COLLIDER_SAFETY` | Hard floor on pairwise chunk distance; zero force above this, exact correction below (no soft falloff tail) |
| `SAFETY_GAP` (per-URL placement) | `5.0` (world units) | Gap added between bounding spheres of successive URL scans: `existing_max_radius + new_radius + SAFETY_GAP` (`force_layout.js` line 63, REAL-02) |
| Camera `minDistance` | `0.6 × cluster_radius(orbit_target)` | Per-frame recomputed orbit-control floor (`projector.md` §5.2; legacy `animation.js` line 794) |
| Camera `maxDistance` | `3.0 × max(\|pos\|)` over all visible nodes | Per-frame recomputed orbit-control ceiling (`projector.md` §5.2; legacy `animation.js` line 765) |
| UMAP tween duration | `~600ms`, eased, interruptible | `chunk_added` → `umap_canonical` canonical-position tween (`projector.md` §4) |
| HSV rotation period | `60s` default, workspace-configurable | Camera-azimuth + chunk-hsv-phase combined rotation period (`projector.md` §5.3) |
| Billboard base size | `1.0` (primary/doc-hub image), `0.55` (secondary/extra image) | Sprite scale before aspect correction (legacy `sprite_manager.js`/`animation.js` line 105) |

Exceptions: none — these are the only "spacing" primitives this phase touches. The surrounding 2D editor's 4/8/16/24/32/48/64 px scale is out of scope (not part of this phase's surface).

## Color

This phase's color contract is **inherited verbatim from `theme.md` §2.5 (the 3D exception zone)** — the one place in the whole application where filled, saturated color is permitted.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) — viewport clear-colour | `#000000` (`--bg-void` / `--black`) | The THREE.js scene background; true black so HSV nodes are maximally legible (`projector.md` §12, `theme.md` §2.1) |
| Secondary (30%) — doc-hub structural elements | `--silver-700` (`#5c616a`) wireframe rings, **steel not coloured** | Per-URL doc-hub instances (`projector.md` §12, `theme.md` §7 table) |
| Accent (10%) — the ONE saturated stroke | `--accent-arrow` `#ffd700` (solid yellow, `stroke-width: 2`) | Reserved EXCLUSIVELY for the 2D↔3D link arrow (`link_layer.md` §3.5, `theme.md` §2.4). Never used for anything else in this phase — not for hover state, not for selection, not for any chrome. |
| Chunk/billboard fill (the exception zone itself) | Backend-computed HSV, `setHSL(h,s,v)` from `umap_canonical` 6-vector channels `[3..5]`, rotated by `(camera_azimuth_phase + chunk_hsv_phase)` | Every chunk node and every image-less billboard. **The frontend renders the backend's HSV; it never invents or computes its own sweep** except as the transient bootstrap fallback (3-vector input → positional `i/n` hue sweep, §11.5/`projector.mjs` lines 41-45) which is discarded the instant a real 6D frame arrives. |
| Image billboard fill | Cached `THREE.Texture` per URL (shared instance across chunks at the same URL) | `REAL-03` — never a flat color when an image is available |
| Destructive | `--accent-error` `#c0414a` | Out of scope for this phase's success criteria (no destructive action lives in the projector itself) — declared here only because the template requires it; not exercised by REAL-01..04 |

Accent reserved for: **the 2D↔3D link arrow stroke, and nothing else.** No hover-highlight color, no selection color, no "active node" tint is introduced — state in the projector is conveyed by position/scale/HSV-rotation only, consistent with `theme.md`'s "state is silver-outline intensity, never colour" rule extended into the 3D exception zone (where the *only* extra color is the arrow).

**Hard-forbidden (CLAUDE.md Forbidden Concepts, do not reintroduce):**
- Any dotted/dashed stroke anywhere (`stroke-dasharray` on any SVG/canvas line) — checked by the existing `black_slate.spec` no-dotted assertion, which REAL-04 must keep green.
- Concentric Fibonacci spheres as a *final* layout position. The hash-direction Fibonacci-style placeholder is permitted **only** as the transient pre-UMAP bootstrap state (`projector.md` §4, "Preliminary radial") and must be superseded by the force-directed/UMAP-locked state once `umap_canonical` arrives.
- Any second saturated hue introduced into the 2D surface as a result of this phase's work (e.g., do not let an image-billboard's dominant color leak a "selection tint" into 2D chrome).

## Phase-Specific Interaction Contracts (codified from CONTEXT + design docs)

These supplement the template's generic sections because this phase's "UI" is 3D geometry and interaction behavior, not forms. Each maps to a REAL-0x requirement.

### REAL-01 — Force-directed ray convergence

- Each chunk's UMAP position fixes a **ray** from its URL's root position through that initial point; thereafter the chunk may move **only along that ray** (1D radial problem per node), never off-ray. (`force_layout.js::_computeRayData`)
- Hard collider repulsion: pairwise distance ≥ `MIN_SEPARATION = 2 × NODE_RADIUS × COLLIDER_SAFETY` exerts **zero force**; below that, exact correction projected onto each node's own ray direction (no soft falloff curve, no spring). (`force_layout.js::_stepForceDirected`)
- **Hard-forbidden as a final position:** concentric/Fibonacci spheres. The Fibonacci-style hash-direction placement is permitted only transiently before the first `umap_canonical` frame lands for a chunk (`projector.md` §4 "Preliminary radial" row).
- Verification anchor: `projector.spec.js` must assert (a) rays exist per chunk-to-root, (b) minimum pairwise spacing holds, (c) no chunk settles at a Fibonacci angular position once `umap_canonical` is applied.

### REAL-02 — Per-URL multi-scan placement + camera framing

- Each URL owns its own `root_position` (`THREE.Vector3`) + `bounding_radius` (scalar, max ray-length of any of its chunks). (`force_layout.js::_initForceLayout`)
- A new URL's root lands at `existing_max_radius + new_radius + SAFETY_GAP` from the nearest existing root, selected by the greedy 12-candidate golden-angle direction search that maximizes minimum distance to all existing roots (`force_layout.js::_computeUrlRootPosition`) — note this candidate-direction search uses the golden-angle sampling **only to choose a placement direction for the root**, which is distinct from (and does not violate) the "no Fibonacci final position" rule, since individual chunk positions are never placed this way — only the one root point per URL.
- Old URLs' roots **never move** on a subsequent scan of a different URL.
- Camera: per-frame-recomputed bounds `minDistance = 0.6 × cluster_radius(orbit_target)`, `maxDistance = 3.0 × max(|pos|)` (`projector.md` §5.2); on scan-end, tween camera to frame/bound the newest root — **suppressed if the user has interacted with the camera AND the root is already in frustum** (`projector.md` §7.1 sequence, REAL-02 acceptance).
- Adaptive resize: both `window.resize` (rAF-coalesced) and a `ResizeObserver` on the projector panel call `setSize(w,h,updateStyle=false)` with **no no-change guard** (`projector.md` §5, "Adaptive resize").

### REAL-03 — Image billboards

- Single-fetch path, checked in this exact order: **in-memory `Map<url, THREE.Texture>` → IndexedDB blob cache → `fetch(proxy_url)` → `fetch(direct_url)`** (`projector.md` §9, `TextureCache`).
- The `X-Image-Proxy-Note` transparent-PNG fallback response is **never cached as a successful image** — a failed/placeholder fetch must not poison the cache.
- Two chunks sharing a URL share **one** `THREE.Texture` instance (no duplicate GPU upload).
- Textures persist for the full layout lifetime — no reload churn on re-render/recompute.
- Billboard scale: base size `1.0` for primary/doc-hub image, `0.55` for secondary/extra images, aspect-corrected (`sprite.scale.set(baseSize*aspect, baseSize, 1)` if `aspect>=1`, else the transpose).
- Collider radius (`NODE_RADIUS = 0.9`) is **shared** between image and text billboards — an image node does not get a different collision footprint than a text node.

### REAL-04 — Solid headless 2D↔3D link arrow

- Every pinned panel that has a 3D-resident node carries a `data-3d-node-id` attribute (existing convention, `concept_graph.js`/`billboard.js`).
- Per animate-loop frame: project that node's world position to screen space via the projector's `project(x,y,z)`; if in-frustum, draw a line from the panel to that screen point; if off-frustum, `display:none` the arrow (no dashed/ghost placeholder).
- Arrow style: **solid**, `stroke="#ffd700"` (`--accent-arrow`), `stroke-width="2"`, **headless** (no `marker-end` arrowhead — undirected per `link_layer.md` §O.16).
- The arrow **drives nothing** — it never moves the panel, never moves the 3D node; it is pure derived render state from `pinned_billboards` + the per-frame projection.
- **Absolute, project-wide hard rule (CLAUDE.md Forbidden Concepts + `theme.md` §8.4):** no `stroke-dasharray` anywhere, on this arrow or on any other line in the 2D plane (hard/soft concept-edge links, compile edges). This phase must not regress the existing `black_slate.spec` no-dotted assertion.

## Assumptions Logged (genuinely unspecified gaps — not blocking)

**A1 — `COLLIDER_SAFETY` numeric value discrepancy.** `docs/frontend/projector.md` §5 prose states `COLLIDER_SAFETY ≥ 2.0`, but the existing reference implementation (`backend/static/js/cp/force_layout.js` line 161) ships `SAFETY = 1.4`. Since this phase explicitly ports the `cp/` reference's *behavior* (not a fresh re-derivation), this UI-SPEC codifies the **shipped value (`1.4`)** as the contract the executor should port, consistent with the phase framing ("bring its features into `fe/`... do not resurrect the surface, port the features/appearance"). If the executor or planner judges the doc's `≥2.0` is the intended target instead, flag it for a one-line doc reconciliation rather than silently picking one — this is a pre-existing doc/code discrepancy from before this phase, not a new decision this phase is introducing.

**A2 — HUD/overlay text inside the projector viewport.** No design doc specifies that any text renders inside the 3D canvas itself (all identity/name display happens via the 2D billboard overlay, a different surface). This UI-SPEC assumes **zero in-canvas text** is added by REAL-01..04, consistent with `projector.md` §10 ("Telemetry... fed from chunks + viewport_visible_rows" — i.e., telemetry lives in the REPL/2D layer, not painted into WebGL). If a future requirement needs in-canvas labels, that is a new decision outside this phase's scope.

**A3 — Dominance-collapse right-click gesture.** `projector.md` §6 lists a pre-existing right-click dominance-collapse/expand gesture on root-URL hubs and bisector nodes. It is not one of REAL-01..04's success criteria and this phase does not need to (re)implement it — it is noted here only so the executor does not accidentally regress it while porting force-layout/billboard/arrow code into the same file.
